Comments & Dispositions

Specification Title: LRMI Terms as RDF
Comment Period: 1-31 December 2014
Draft Comment Text: http://dublincore.org/dcx/lrmi-terms/drafts/2014-11-30/
Comments to: lrmi-comment(at)dublincore.net


ID Date Source Comments Editors' Comments Task Group Disposition
1 2014-12-10 Ervin Flack One editorial comment: In the last row of the first table under 1. Introduction & Definitions
"Date Issued/Modified: The type of term was created or modified." [s]hould read "Date Issued/Modified: Date the term was created or modified".		 This is a transcription error and will be corrected. Error corrected
2a 2015-01-08 Douglas It's unfortunate that some descriptions talk generically about the 'resource' or 'content' whereas others are more constraining by specifying 'learning resource' only. But it sounds like that cannot be changed due to the verbatim transfer requirement.
(https://www.jiscmail.ac.uk/cgi-bin/webadmin?A2=DC-ARCHITECTURE;d38d0f8e.1501)		 Douglas is correct that these variations are, with the exception of the lrmi:EducationalResource definition, transcribed verbatim from the approved LRMI 1.1. --
2b 2015-01-08 Douglas However, it appears that the descriptions for the classes are newly written (not in the LRMI specs) so could be made more generic. I can see EducationalAudience being very useful in wider contexts so it could refer to a generic resource rather than a learning resource.
(https://www.jiscmail.ac.uk/cgi-bin/webadmin?A2=DC-ARCHITECTURE;d38d0f8e.1501)		 The lrmi:EducationalAudience is the only property or class with a new draft definition because there was none provided initially in version 1.1. The EducationalAudience class in schema.org merely states that it is "based on the work of the LRMI project" (http://schema.org/EducationalAudience). LRMI merely states that the class is a subclass of schema:Audience (http://www.lrmi.net/the-specification). While each context names the class and points to the other, neither schema.org nor LRMI define the class. So, this is indeed a draft that could be revised without violating the agreement that 1.1 is adopted as-is. --
2c 2015-01-08 Douglas And in fact, I'm realising the description [lrmi:EducationalAudience] should probably describe the audience explicitly rather than it being inferred because the resource is a learning resource. So maybe something like "A group the resource was created for, or is useful to, in an educational context."? Or along the lines of the dc:audience description "A group for whom the resource is intended or useful in an educational context".
(https://www.jiscmail.ac.uk/cgi-bin/webadmin?A2=DC-ARCHITECTURE;d38d0f8e.1501)		 -- --
2d 2015-01-08 Douglas My other thought is: should there be more links to DCMI terms? e.g. educationalAlignment, isBasedOnURL, etc. sub-classing dct:relation, educationalRole sub-classing dct:audience, and is useRightsURL necessary when dct:rights seems to do the same thing?
(https://www.jiscmail.ac.uk/cgi-bin/webadmin?A2=DC-ARCHITECTURE;d38d0f8e.1501)		 The TG should consider all appropriate sub-property/sub-class references to dcterms (http://dublincore.org/specifications/). --
3 2015-01-08 Adrian Pohl I noticed that some property URIs aren't correct in the LRMI comment draft and thus don't resolve:

- http://schema.org/isBasedOnURL -> http://schema.org/isBasedOnUrl
- http://schema.org/targetURL -> http://schema.org/targetUrl

(https://www.jiscmail.ac.uk/cgi-bin/webadmin?A2=DC-ARCHITECTURE;d8c6f808.1501)		 These are transcription errors and will be corrected. Errors corrected
4 2015-01-10 Stuart Sutton The definition of lrmi:typicalAgeRange is grammatically incorrect. Shouldn't "The typical range of ages the content's intended end user" be something like "The typical range of ages of the content's intended end user"? This would be a non-substantive change to the property definition (see §3A of DCMI Namespace Policy at http://dublincore.org/documents/dcmi-namespace/index.shtml on correcting 'minor editorial errata'). --
5 2015-01-10 Stuart Sutton There seems to be a URL mistakenly included in the description of the lrmi:isBasedOnUrl that should be struck: "This term may be repeated for multiple resources. For example, http://example.com/great-multiplication-intro.html" Removed in 2015-01-12 draft iteration (http://dublincore.org/dcx/lrmi-terms/drafts/2015-01-12/). Error corrected.
5 2015-01-13 Stuart Sutton There is an inconsistent use throughout of rdfs:domain/rdfs:range and the schema.org schema:domainIncludes and schema:rangeIncludes properties in the 2014-11-30 draft iteration (http://dublincore.org/dcx/lrmi-terms/drafts/2014-11-30/). In keeping with LRMI's roots in the schema.org commitment to looser ontological commitment, all rdfs:domain and rdfs:range declarations in the 2014-11-30 draft have been changed in the 2015-01-12 draft iteration to schema:domainIncludes and schema:rangeIncludes (http://dublincore.org/dcx/lrmi-terms/drafts/2015-01-12/) --


Not included above in the Comments & Dispositions table is an exchange on the DC-Architecture jiscmail forum regarding the creation of the RDF serialization. DCMI by policy can't currate the LRMI specification without a formal schema in a DCMI controlled namespace. Since 2000 all terms in DCMI namespaces have been declared in RDF.
